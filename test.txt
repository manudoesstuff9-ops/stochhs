# Where do configs come in?

This is the part I would do **differently** from what you suggested.

I **wouldn't** start by filling in every YAML file.

Instead:

1. Create the YAML files now (empty or minimal).
    
2. Ignore them while writing the first version of each module.
    
3. Once a module works, move its hardcoded parameters into the corresponding config file.
    

For example, don't start with:

```yaml
particle_filter:
  particles: 5000
  resampling: systematic
  ess_threshold: 0.5
```

Instead, write the particle filter first:

```python
pf = ParticleFilter(
    num_particles=1000,
    resampling="systematic",
)
```

Once you're happy with it, refactor it to:

```python
cfg = load_config("particle_filter.yaml")
pf = ParticleFilter(**cfg)
```

This avoids spending hours designing configuration before you know what the code actually needs.

```
        │

        │
        ▼
Pipeline Automation
        │
        ▼
Documentation & README
```

I think this order is much more realistic because **every new layer has something concrete to consume**. You're never writing algorithms in isolation—they always plug into the layer beneath them. By the time you reach `run_pipeline.py`, you're simply connecting components that already work, rather than trying to invent the whole architecture at once. That tends to produce a much more maintainable research codebase.
